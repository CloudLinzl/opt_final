# 接口规范

## 一、文档目的

本规范用于统一四位成员在迷宫路径规划项目中的地图输入、算法输出、实验场景与评判标准，保证不同算法可以在同一批数据上运行，并直接汇总比较结果。

适用算法包括：

1. BFS
2. DFS
3. A*
4. Cost-aware A*
5. 改进 A* / 带权 A*
6. LPA*
7. D* Lite

---

## 二、地图数据来源

为保证实验数据真实、可复现，地图原则上优先来源于历年 Micromouse 竞赛公开迷宫。

统一约定如下：

1. 以标准 `16 x 16` 迷宫为主；
2. 所有算法默认使用同一批原始地图；
3. 动态算法不单独更换地图，而是在同一底图上加入动态障碍更新；
4. 每张地图必须有唯一编号，便于实验记录与结果对照。

建议统一命名方式：

`maze_01`, `maze_02`, `maze_03` ...

---

## 三、统一地图输入格式

每张地图至少包含以下字段：

| 字段 | 类型 | 含义 |
| ---- | ---- | ---- |
| `maze_id` | string | 地图编号 |
| `width` | int | 地图宽度，默认 16 |
| `height` | int | 地图高度，默认 16 |
| `wall_matrix` | array | 墙体信息或邻接阻塞信息 |
| `start` | array | 起点坐标，如 `[x, y]` |
| `goal` | array | 终点坐标，如 `[x, y]` |
| `goal_cells` | array | 目标区域坐标集合；若只有单目标，可只包含一个坐标 |
| `risk_cells` | array | 风险区域坐标集合，可为空 |
| `narrow_cells` | array | 狭窄区域坐标集合，可为空 |
| `dynamic_updates` | array | 动态障碍更新信息，可为空 |

推荐约定：

1. 坐标系统一使用左上角为原点；
2. `x` 表示列坐标，`y` 表示行坐标；
3. 所有成员使用相同坐标方向，避免路径结果无法对齐；
4. `wall_matrix` 的编码方式全组统一即可，推荐使用“每个格子四个方向是否有墙”的结构；
5. 对于 Micromouse 标准迷宫，中心目标通常为 `2x2` 区域，因此建议同时保留 `goal` 与 `goal_cells` 两个字段，其中 `goal` 可取 `goal_cells` 中排序后的第一个坐标，供只支持单目标的算法兼容使用。

---

## 四、动态更新字段规范

`dynamic_updates` 建议统一为如下结构：

| 字段 | 类型 | 含义 |
| ---- | ---- | ---- |
| `step` | int | 在第几步或第几个时刻触发更新 |
| `blocked_cells` | array | 新增障碍的坐标集合 |
| `released_cells` | array | 解除封锁的坐标集合，可为空 |

示例：

```json
[
  {
    "step": 20,
    "blocked_cells": [[5, 6], [5, 7]],
    "released_cells": []
  }
]
```

说明：

1. `BFS` 与基础 `A*` 可不读取该字段；
2. `LPA*` 与 `D* Lite` 必须支持该字段；
3. 若无动态更新，则统一写空数组 `[]`。

---

## 五、统一主流程约定

当前项目主流程由 `src/main.py` 负责，默认处理顺序如下：

1. 读取一张静态 JSON 地图；
2. 调用已注册的静态算法；
3. 在主流程中生成动态事件与动态图文件；
4. 将 `dynamic_updates` 传入 `LPA*` 与 `D* Lite`；
5. 统一保存算法结果 JSON；
6. 调用统一可视化接口并保存产物清单。

第一版默认只处理单张静态地图，不做批量实验入口。
主流程支持通过 `--algorithms` 指定算法子集，只运行需要的算法键。

## 六、统一算法调用接口

后续所有算法模块都应尽量遵守统一调用风格：

| 算法类型 | 推荐函数签名 |
| ---- | ---- |
| 静态算法 | `run_xxx(map_input, **kwargs) -> dict` |
| 动态算法 | `run_xxx(map_input, dynamic_updates=None, **kwargs) -> dict` |

推荐示例：

```python
run_bfs(map_input, **kwargs) -> dict
run_a_star(map_input, **kwargs) -> dict
run_cost_aware_a_star(map_input, **kwargs) -> dict
run_weighted_a_star(map_input, **kwargs) -> dict
run_lpa(map_input, dynamic_updates=None, **kwargs) -> dict
run_dstar_lite(map_input, dynamic_updates=None, **kwargs) -> dict
```

其中：

1. `map_input` 可以是地图路径，也可以是已经加载好的 `map_data`；
2. 动态算法优先支持外部传入 `dynamic_updates`，避免强依赖预生成动态图文件；
3. 返回值必须对齐本规范中的统一结果字段；
4. 若算法模块尚未实现，主流程允许返回占位结果，但字段结构不能缺失。

| 算法 | 必需字段 | 可选字段 |
| ---- | -------- | -------- |

| LPA* | 上述全部基础字段 | `dynamic_updates` |
| D* Lite | 上述全部基础字段 | `dynamic_updates` |

## 八、统一算法输出格式

所有算法运行后，统一输出以下字段：

| 字段 | 类型 | 含义 |
| ---- | ---- | ---- |
| `maze_id` | string | 当前测试地图编号 |
| `algorithm` | string | 算法名称 |
| `success` | bool | 是否成功找到路径 |
| `path` | array | 路径坐标序列 |
| `path_length` | float/int | 路径长度 |
| `turn_count` | int | 转弯次数 |
| `risk_cost` | float/int | 路径经过风险区域的累计代价 |
| `narrow_cost` | float/int | 路径经过狭窄区域的累计代价 |
| `explored_nodes` | int | 搜索或扩展节点数 |
| `explored_cells` | array | 已扩展节点坐标集合，供可视化使用 |
| `runtime_ms` | float | 运行时间，单位毫秒 |
| `total_cost` | float | 综合代价 |
| `visualization_trace` | array | 过程可视化帧序列 |
| `heuristic_weight` | float/null | 启发函数权重；非加权算法可为空 |
| `cost_weights` | object/null | 当前算法使用的代价系数配置 |

对于动态算法，再额外输出：

| 字段 | 类型 | 含义 |
| ---- | ---- | ---- |
| `replan_count` | int | 重规划次数 |
| `replan_time_ms` | float | 单次或平均重规划时间 |
| `updated_nodes` | int | 动态更新中受影响节点数 |
| `replanned_path_length` | float/int | 更新后路径长度 |

约定：

1. `BFS` 与基础 `A*` 中，若没有风险区、狭窄区和动态更新，则 `risk_cost=0`、`narrow_cost=0`、`replan_count=0`；
2. 若算法失败，`success=false`，其他字段可写空值或 `-1`，但全组必须统一；
3. `runtime_ms` 统一使用毫秒；
4. `path` 必须能还原完整路径，便于后续可视化；
5. `total_cost` 必须可由本规范中的统一公式反推出；
6. 若算法支持过程动画，则 `visualization_trace` 不应缺失。

| 字段 | 含义 |

| `artifact_type` | 产物类型 |

## 十一、时间与代价计算规范

### 11.1 运行时间计算方法

为保证不同成员的实验结果可以比较，统一采用如下计时规则：

1. **只统计算法核心求解时间**
   不计入地图文件读取、数据打印、绘图显示、动画播放、结果保存等额外时间。
2. **计时起点**
   在地图数据已经载入内存、起点终点与附加字段已经准备完成后，算法主函数正式开始搜索的瞬间开始计时。
3. **计时终点**
   在算法返回最终路径，或明确判定“无可行路径”时结束计时。
4. **静态算法的 `runtime_ms`**
   指单次完整求解时间。
5. **动态算法的 `runtime_ms`**
   指“首次规划时间 + 所有重规划时间”的总和。
6. **动态算法的 `replan_time_ms`**
   指单次重规划的平均时间，计算方式为：

\[
replan\_time\_ms=
\begin{cases}
\dfrac{\text{all replan time}}{replan\_count}, & replan\_count>0 \\
0, & replan\_count=0
\end{cases}
\]

7. **重复实验规则**
   每张地图每种算法连续运行 `10` 次，去掉最大值和最小值后，对剩余 `8` 次求平均，作为最终 `runtime_ms`。
8. **实验环境要求**
   尽量在同一台设备、同一语言环境、相近负载条件下测试，避免后台程序波动影响结果。

---

### 11.2 转弯次数 `turn_count` 的统一定义

设路径为：

\[
P=(v_0,v_1,\dots,v_k)
\]

其中相邻节点之间的移动方向记为：

\[
d_i=v_i-v_{i-1},\quad i=1,2,\dots,k
\]

则转弯代价按以下规则计算：

1. 若 \(d_i=d_{i-1}\)，记为直行，不增加转弯代价；
2. 若 \(d_i\) 与 \(d_{i-1}\) 为左转或右转，记 `1` 次转弯；
3. 若 \(d_i\) 与 \(d_{i-1}\) 方向相反，即发生掉头，记 `2` 次转弯。

因此统一定义：

\[
turn\_count = N_{90} + 2N_{180}
\]

其中：

* \(N_{90}\) 表示 90 度转弯次数；
* \(N_{180}\) 表示 180 度掉头次数。

说明：

1. 对于正常最短路算法，掉头通常较少出现；
2. 若实际实现中禁止掉头，则 \(N_{180}=0\)；
3. 全组必须使用同一转弯统计方式，不能有人把掉头记 1 次、有人记 2 次。

---

### 11.3 风险代价与狭窄代价定义

为便于统一计算，默认采用最简单、最容易复现的离散累加方式。

设最终路径经过的节点集合为 \(P\)，则：

\[
risk\_cost=\sum_{v \in P} r(v)
\]

\[
narrow\_cost=\sum_{v \in P} q(v)
\]

其中：

* 若节点 \(v\) 属于 `risk_cells`，则默认 \(r(v)=1\)，否则 \(r(v)=0\)；
* 若节点 \(v\) 属于 `narrow_cells`，则默认 \(q(v)=1\)，否则 \(q(v)=0\)。

若后续希望增强区分度，也可扩展为分级代价，例如：

* 高风险区域：\(r(v)=2\)
* 中风险区域：\(r(v)=1\)
* 普通区域：\(r(v)=0\)

但在本阶段，为了方便四位成员统一实现，默认先使用二值代价。

---

### 11.4 `total_cost` 的统一计算方式

为避免硬件差异直接影响综合代价，统一规定：

**`runtime_ms` 只作为效率评价指标，不直接并入 `total_cost`。**

因此，所有算法统一使用如下总代价公式：

\[
total\_cost = p_1 \cdot path\_length + p_2 \cdot turn\_count + p_3 \cdot risk\_cost + p_4 \cdot narrow\_cost + p_5 \cdot replan\_count
\]

其中：

* `path_length`：路径长度
* `turn_count`：转弯代价
* `risk_cost`：风险区域累计代价
* `narrow_cost`：狭窄区域累计代价
* `replan_count`：动态场景中的重规划次数

对静态算法而言：

\[
replan\_count = 0
\]

因此静态算法可视为：

\[
total\_cost = p_1 \cdot path\_length + p_2 \cdot turn\_count + p_3 \cdot risk\_cost + p_4 \cdot narrow\_cost
\]

---

### 11.5 系数 \(p_i\) 的参考含义与建议取值

统一规定 `p_1` 作为路径长度的基准权重，其余系数相对它进行设定。

| 系数 | 含义 | 参考范围 | 默认建议值 | 解释 |
| ---- | ---- | -------- | ---------- | ---- |
| \(p_1\) | 单位路径长度代价 | `1` | `1.0` | 作为基准项 |
| \(p_2\) | 单位转弯代价 | `0.5 ~ 2.0` | `1.0` | 表示一次 90 度转弯大约等价于多走 1 步 |
| \(p_3\) | 单位风险区域代价 | `2.0 ~ 5.0` | `3.0` | 用于体现“宁可稍远，也要避开高风险区域” |
| \(p_4\) | 单位狭窄区域代价 | `1.0 ~ 3.0` | `2.0` | 体现对狭窄通道的适度规避 |
| \(p_5\) | 单次重规划代价 | `1.0 ~ 4.0` | `2.0` | 用于动态算法中惩罚频繁重规划 |

推荐的默认总代价公式为：

\[
total\_cost = 1.0 \cdot path\_length + 1.0 \cdot turn\_count + 3.0 \cdot risk\_cost + 2.0 \cdot narrow\_cost + 2.0 \cdot replan\_count
\]

该组参考值体现如下偏好：

1. 路径长度仍然是最基本目标；
2. 转弯需要惩罚，但不应比风险规避更重要；
3. 风险区惩罚高于狭窄区惩罚；
4. 动态算法若频繁重规划，说明路径稳定性较差，应给予一定惩罚。

若后续实验发现某一项影响过强或过弱，可以在全组统一讨论后调整，但必须保证：

1. 所有成员使用同一组 \(p_i\)；
2. 报告中明确写出最终采用的系数；
3. 调参前后的实验结果分开记录，避免混淆。

## 十二、可直接贴入正文的参数说明

本文在综合代价设计中，采用路径长度、转弯代价、风险区域代价、狭窄区域代价与重规划次数五项指标共同评价路径质量。设最终综合代价为 `total_cost`，则统一定义为

\[
total\_cost = p_1 \cdot path\_length + p_2 \cdot turn\_count + p_3 \cdot risk\_cost + p_4 \cdot narrow\_cost + p_5 \cdot replan\_count
\]

其中，`path_length` 表示路径长度，`turn_count` 表示转弯次数，`risk_cost` 表示路径经过风险区域的累计代价，`narrow_cost` 表示路径经过狭窄区域的累计代价，`replan_count` 表示动态环境中的重规划次数。考虑到不同设备的运行性能差异，`runtime_ms` 不直接并入 `total_cost`，而作为独立效率指标进行统计与比较。为保证实验初期的统一性，本文暂取 `p_1=1.0`、`p_2=1.0`、`p_3=3.0`、`p_4=2.0`、`p_5=2.0` 作为参考值，即默认认为一次 90 度转弯约等价于多走 1 步，经过一次风险区域的代价高于经过一次狭窄区域的代价，而频繁重规划会额外增加路径总代价。上述系数在后续实验中可统一微调，但同一轮对比实验必须保持一致。

## 十三、统一实验场景

全组统一使用以下三类实验场景：

### 13.1 基础通关场景

特点：

1. 地图静态；
2. 所有可行边代价一致；
3. 只考察最短路与搜索效率。

适用算法：

1. BFS
2. A*

### 13.2 高风险绕行场景

特点：

1. 地图仍为静态；
2. 部分区域设置 `risk_cells`；
3. 部分区域设置 `narrow_cells`；
4. 需要综合考虑路径长度、转弯和风险。

适用算法：

1. 基础 A*
2. 改进 A* / 带权 A*

### 13.3 突发封路场景

特点：

1. 地图初始可行；
2. 运行过程中触发 `dynamic_updates`；
3. 算法需要在中途快速更新路径。

适用算法：

1. LPA*
2. D* Lite

## 十四、统一评判标准

统一指标如下：

| 指标 | 说明 |
| ---- | ---- |
| 路径长度 | 是否缩短移动距离 |
| 转弯次数 | 是否提高运行平稳性 |
| 综合代价 | 是否兼顾距离、转弯和风险 |
| 搜索节点数 | 是否减少无效扩展 |
| 运行时间 | 是否提升计算效率 |
| 重规划时间 | 动态变化后的恢复速度 |
| 成功率 | 是否稳定找到可行路径 |

进一步约定：

1. `BFS` 与基础 `A*` 重点比较：路径长度、搜索节点数、运行时间；
2. 改进 `A*` 重点比较：综合代价、转弯次数、风险规避能力；
3. `LPA*` 与 `D* Lite` 重点比较：重规划时间、更新节点数、路径恢复效果。

## 十五、统一结果总表建议

建议最后全组汇总成统一结果表，表头如下：

| maze_id | algorithm | success | path_length | turn_count | risk_cost | narrow_cost | explored_nodes | runtime_ms | total_cost | replan_count | replan_time_ms | updated_nodes | replanned_path_length |
| ------- | --------- | ------- | ----------- | ---------- | --------- | ----------- | -------------- | ---------- | ---------- | ------------ | -------------- | ------------- | --------------------- |

说明：

1. 静态算法没有动态字段时，可统一填 `0` 或空值；
2. 所有成员提交结果时，表头顺序保持一致；
3. 最终报告和 PPT 直接从这张总表生成统计图和对比图。

## 十六、默认实现约定

为避免后续重复沟通，默认做如下约定：

1. 地图尺寸默认 `16 x 16`；
2. 起点和终点坐标由全组统一设定；
3. 所有算法尽量使用同一语言与相似数据结构；
4. 时间统计统一采用同一种计时方式；
5. 结果表中的空值写法由全组统一一次后不再更改；
6. 主流程优先通过统一注册接口调用算法模块，而不是在 `main` 中硬编码算法细节；
7. 动态图生成由 `main` 触发，但应同时保存动态图文件，保证实验可复现。

如果后续代码实现中有额外字段，可新增，但不得删改本规范中的基础字段。
